# Фундамент под AI: что нужно построить, прежде чем доверять проект AI

*Для технических специалистов — DevOps, BA, PM, SM — которые начинают работать с агентными AI: Claude Code, GitHub Copilot, Codex*

---

## Layer 0. Context Rot: когда AI начинает тупить

Представьте: вы открываете новую сессию с AI-ассистентом и просите продолжить работу над фичей, которую начали вчера. Модель умная, контекстное окно огромное — но через несколько сообщений вы замечаете, что она не помнит принятое вчера архитектурное решение, предлагает структуру файлов, которую вы уже отвергли, и повторяет ошибки из прошлой сессии. Знакомо?

Вот как часто это выглядит в работе с AI:

  Три дня назад архитект обновил спецификацию API — поменялась структура ответа, добавилось поле `refresh_token`. Изменения лежат в Confluence. Агент пишет старую схему — ту, что видел в начале сессии. Код компилируется, тесты зелёные. На интеграционном стенде — провал.

  Другой сценарий: команда работает на TypeScript, внутренние правила строгие — никаких `any`, функции не длиннее 50 строк, обязательный error handling. Но об этом никто не сказал агенту: нет ни файла, ни конфига, ни правил "чистого кода". Агент пишет рабочий код — `any` везде, функции по 200 строк, исключения проглочены. Code review — провал. Переписывать руками.

  Если вы преодолели сценарии выше то вас ждет: два агента работают параллельно. Один добавляет хелпер авторизации в `utils.ts`, второй рефакторит те же утилиты. Merge-конфликт — и оба изменения при этом логически правильные. Просто не было их координации.

  Наконец — к середине длинной сессии контекстное окно забито ранними итерациями, промежуточными вариантами, дискуссиями о деталях реализации. Модель начинает путаться и противоречить сама себе.

Всё это называют **context rot** — *«гниение контекста»*: деградация качества работы агента по мере накопления нерелевантного или устаревшего контекста. Первый инстинкт — взять модель с окном побольше. Не поможет. Исследования показывают: больше контекста без организации усугубляет проблему, потому что модель не умеет сама отличать важное от шума.

Нужна не умнее модель. Нужна **дорога** — а не лучший водитель.

## Layer 1. Context Engineering: итак, как строить дорогу?

В 2025 году Gartner зафиксировал смену парадигмы: **«Context engineering is in, prompt engineering is out»** (Hype Cycle for Artificial Intelligence, 2025). Разница принципиальная.

**Prompt engineering** — это инструкция водителю для одной конкретной поездки. **Context engineering** — проектирование всей транспортной инфраструктуры: *что* попадает в контекстное окно агента, *когда*, *в какой форме* и *что остаётся вынесенным за его пределы*. Если промпт-инжиниринг работает с одним запросом, то контекст-инжиниринг решает задачу другого масштаба — как сделать так, чтобы агент сохранял точность и последовательность на протяжении длинного, многосессионного, параллельного проекта.

![Эволюция от промптов к автономным агентным циклам. Слева: традиционный ассистент — один запрос, один ответ. Справа: агентный SDK с циклом планирования, оркестрации инструментов, оценки и итеративного уточнения.](Illustrations/From%20prompt%20to%20autonomous%20loops.jpg)

> **Context Engineering** — дисциплина управления информационным потоком в LLM-системах: определяет, какие данные, в каком объёме и в какой момент попадают в контекстное окно агента, чтобы обеспечить точность и предсказуемость на каждом шаге рабочего процесса.

Цель — не загрузить максимум, а найти **минимально необходимый и достаточный контекст** для следующего шага. На практике этот принцип раскладывается на шесть операций — и каждая из них закрывает один из сценариев сбоя, которые мы только что разобрали.

### Принцип 1. Context Offloading — файловая система как внешняя память

**Context Offloading** — перенос всего, что должно пережить сессию, из контекстного окна в постоянное хранилище: файлы и git.

Звучит банально. На практике это означает строгое правило: архитектурное решение не существует, пока не записано — не в чат, а в файл, сразу, не «потом в конце сессии». В шаблоне за это отвечают `planning-artifacts/decisions.md` — живой лог всех архитектурных решений, `project-status.md` — текущее состояние проекта, и `session-context.md` — сжатые резюме предыдущих сессий.

Особого внимания заслуживают **git micro-commits** — контрольные точки на каждое значимое изменение в формате `[T-{id}] что изменено`. Для BA и PM это принципиально: если что-то пошло не так, система не откатывается к нулю, а возвращается к последней стабильной точке. Git-история здесь — не просто версионирование, а встроенная страховочная сеть.

Правило одно: **решение, которое не записано в файл, не существует**.

![Использование файловой системы как контекста. Слева: антипаттерн — весь контекст растёт внутри окна. Справа: контекст содержит только ссылки, детали читаются из файлов по необходимости.](Illustrations/Use%20file%20system%20as%20context.png)

### Принцип 2. Context Retrieval — читать актуальное, а не помнить старое

Offloading решает проблему сохранности. Но что, если агент читает не то, что актуально? Именно здесь вступает **Context Retrieval** — чтение текущего состояния из файлов в начале каждого рабочего цикла вместо опоры на то, что «было в памяти».

Stateless-диспетчер, который начинает каждый цикл с чтения TaskList и последних артефактов, никогда не работает на устаревших данных. Никакого состояния в памяти — только то, что прочитано прямо сейчас. Именно это закрывает первый сценарий из Layer 0: агент видит актуальную спецификацию с `refresh_token`, а не ту, что была при старте сессии.

Исследовательский агент следует **multi-layer RAG** стратегии с жёсткой приоритетной цепочкой:

```
1. Существующие артефакты исследования (< 7 дней) — не искать заново
2. knowledge-base/ — локальная папка с кураторскими файлами
3. Context7 MCP — документация библиотек и фреймворков
4. Tavily MCP — RAG-оптимизированный веб-поиск
5. Прямой веб-поиск — только если всё выше не дало ответа
```

Стоит выделить два момента. Во-первых, `knowledge-base/` — это обычная папка: бросьте туда PDF-статью, экспорт из Confluence или фрагмент документации, и агент найдёт её на шаге 2, не расходуя токены на веб-поиск. Ваша собственная база знаний для агентов. Во-вторых, Tavily — в отличие от обычного поиска — суммаризирует результаты на своей стороне и возвращает только компактный дайджест, а не полные HTML-страницы. Агент получает готовый ответ, не «пережёвывая» лишний контент. В совокупности стратегия снижает потребление токенов на **60–80%** по сравнению с прямым веб-поиском.

### Принцип 3. Context Reduction — токенный бюджет и декомпозиция

Даже при правильном offloading и retrieval контекст может перегружаться — просто за счёт объёма накопленной работы. **Context Reduction** — управление этим объёмом через лимиты, автоматическую компакцию и маршрутизацию задач по сложности.

Целевой лимит: главный агент удерживается в пределах **128k токенов** на весь проект. При достижении **80k** запускается компакция: старые диалоги сжимаются в JSON-резюме, а критические ссылки — например, сжатые стандарты кодирования — никогда не вытесняются из контекста. Кроме того, не каждая задача требует самой мощной модели: простые запросы идут к дешёвой быстрой модели, сложная архитектура — к мощной. Это не экономия, а дисциплина: правильный инструмент для правильной задачи.

### Принцип 4. Context Isolation — каждый специалист в своём контексте

**Context Isolation** — разделение агентов по ролям так, чтобы каждый получал только контекст, необходимый для его задачи. Один «универсальный» агент, который делает всё, к середине проекта несёт в памяти и детали авторизации, и структуру базы данных, и результаты code review трёхнедельной давности. Точность падает. Изоляция — единственный способ с этим справиться. Подробнее — в следующем разделе.

### Принцип 5. Context Orchestration — единый диспетчерский цикл

**Context Orchestration** — управление всеми стадиями через единый предсказуемый **dispatch loop**:

```
Читать состояние → Выбрать задачу → Подобрать агента
    → Классифицировать сложность → Диспетчеризовать
        → Записать результат → Повторить
```

Никаких импровизаций. Один паттерн — меньше сюрпризов, проще отладка. Важная деталь: каждый диспатч получает уникальный **Trace ID** в формате `TRACE-{дата}-{время}-{3-слова}`. Каждый артефакт — план, код, ревью, тест-отчёт — несёт этот идентификатор. Если что-то пошло не так, всегда можно восстановить полную цепочку: какой агент, по какой задаче, какое решение принял. Встроенная наблюдаемость — не дополнение к системе, а часть её ядра.

### Принцип 6. Context Governance — качество, безопасность и предохранители

**Context Governance** — протоколы, ограничения и автоматические проверки, которые обеспечивают качество и безопасность на каждом шаге.

Первое: **инструменты только из верифицированных источников**. В шаблоне действует иерархия доверия: локальные переопределения команды → проверенные стандарты → внешние источники сообщества. Агент не получает доступ к инструменту или источнику, который не прошёл через эту иерархию — это защита от галлюцинаций на уровне инструментов, а не на уровне промпта.

Второе: **auto-feedback loop** — система сама себя проверяет в рамках заданного числа итераций. Reviewer следует структурированному протоколу: явные статус-коды, пронумерованные проблемы с рейтингом серьёзности, конкретные инструкции по исправлению. Максимум три цикла — **circuit breaker**. Если за три итерации задача не закрыта — эскалация к человеку. Не четвёртый цикл.

Третье: **накопление памяти об ошибках**. После каждого review-цикла Reviewer проверяет, нашёл ли он паттерны ошибок, которых не было раньше. Если да — дописывает их в `failure-patterns.md`. Все последующие агенты читают этот файл перед началом работы. Одна и та же архитектурная ловушка не повторяется дважды — она уже зафиксирована и известна.

Наконец, четвёртое: **три системных предохранителя** — каждые 10 задач система проверяет себя:
- *Cascade failure*: больше трёх задач одновременно BLOCKED → диспатч приостанавливается, пользователь получает сводку
- *Dispatch loop*: одна задача диспатчирована больше пяти раз → блокировка с полной историей
- *Plan explosion*: одно планирование создало больше 15 задач → флаг для ревью до запуска

Система не замалчивает системные проблемы — она их обнаруживает и поднимает. Плюс защита от утечек секретов на трёх уровнях: правила `.gitignore`, ограничения агентов и автоматическое сканирование коммитов.

## Layer 2. Multi-Agent Architecture: когда одного AI недостаточно

Представьте команду из одного человека, который одновременно является аналитиком, архитектором, разработчиком, тестировщиком и ревьюером. Формально справляется. По факту — постоянно теряет фокус из-за переключения ролей. С AI-агентами ровно та же история: один «умный на всё» агент неизбежно накапливает нерелевантный контекст и теряет точность.

![Архитектура главный агент — субагенты. Главный агент формирует инструкцию; субагент получает свой системный промпт, инструменты и изолированный контекст. Результат возвращается наверх.](Illustrations/For%20each%20subagent%20make%20instruction.png)

В шаблоне шесть специализированных агентов — каждый получает только то, что нужно для своей роли:

| Агент | Роль | Что получает в контекст |
|---|---|---|
| **Researcher** | Поиск, оценка технологий | Вопрос + локальный кеш |
| **Planner** | Разбивка на задачи, зависимости | Цели + текущий статус |
| **Architect** | Системный дизайн, решения | Принципы + лог решений |
| **Implementer** | Написание кода | Спецификация + coding standards |
| **Reviewer** | Code review | Код + чеклист + спецификация |
| **Tester** | Запуск тестов, верификация | Критерии готовности + scope |

Каждый агент — отдельный `.md`-файл. Добавить нового специалиста — создать файл. Система обнаружит его автоматически.

### Delegated Mechanism Selection

Ключевой принцип, который стоит назвать явно: **главный агент решает *что сделать* и *кому поручить*, субагент решает *как именно***.

Это **Delegated Mechanism Selection**: механизм исполнения делегируется тому, кто лучше знает свою область. Главный агент не знает и не должен знать, какой инструмент использует Researcher для поиска или как именно Tester запускает тесты. Он знает только *что нужно* и *кому это поручить*. Такое разделение — основа масштабируемости: добавление нового агента не требует изменения оркестратора.

![Иерархия инструментов агентов. Уровень 1: атомарные операции (~20 инструментов, стабильны). Уровень 2: утилиты sandbox. Уровень 3: сложная логика через скрипты и библиотеки.](Illustrations/Hierarchical%20Action%20Space%20for%20limiting%20amount%20of%20agents.jpg)

Иерархия инструментов — это Context Isolation на уровне возможностей. Implementer имеет доступ к редактированию файлов и git, Researcher — к поиску и кешам, Reviewer — только чтение. Агент буквально не может вызвать инструмент, к которому у него нет доступа. Это структурное ограничение, а не просьба в промпте — принципиальная разница.

**War story — параллельная работа.** Два агента ведут параллельные фичи. Planner до диспатча анализирует зависимости файлов и убеждается, что области работы не пересекаются. Агенты запускаются одновременно. Merge-конфликт исключён архитектурно — не надеждой на осторожность агента, а структурным ограничением. Это **parallel fan-out**: несколько агентов параллельно, но только на непересекающихся файлах. Вся реализация — на feature-ветках, никогда напрямую на `main`.

### Что если агент обнаружил, что задача больше, чем казалось?

Субагент не только выполняет задачу — он имеет право её остановить. Если Implementer в процессе работы обнаруживает, что задача значительно сложнее первоначальной классификации (например, требует изменений в восьми файлах вместо трёх), он обязан: СТОП → написать ESCALATED-артефакт с рекомендацией (RESPEC / SPLIT / ARCHITECT_REVIEW) → заблокировать задачу. Dispatch loop переклассифицирует и перемаршрутизирует. Максимум одна эскалация на задачу.

Назовём это **autonomous boundary recognition**: агент не прорывает границы задачи вслепую и не тихо раздувает scope — он сигнализирует, когда границы были определены неверно. Для PM это означает: неожиданная сложность не приведёт к тихому созданию непредвиденных изменений по всей кодовой базе.

## Layer 3. Spec-Driven Development (SDD): что значит «готово»?

CE отвечает на вопрос *как работать*. SDD отвечает на вопрос *когда остановиться*.

Implementer считает задачу выполненной, когда код компилируется. Reviewer — когда нет критических замечаний. Tester — когда тесты зелёные. Planner — когда выполнены все подзадачи. Все правы по-своему — но фича при этом не делает того, что нужно пользователю. Это системная проблема мультиагентных систем, и CE сама по себе её не решает.

> **Spec-Driven Development (SDD)** — подход, при котором каждая фича получает машиночитаемую спецификацию с верифицируемыми критериями готовности *до* начала реализации. Спецификация существует в файлах и переживает любую потерю контекста.

Каждая фича получает **spec packet** — структурированный YAML-документ, встроенный прямо в описание задачи:

```yaml
# --- SPEC ---
version: 1
intent: "Return JWT token on valid credentials, reject on invalid"
assertions:
  - id: A1
    positive: "POST /auth with valid credentials MUST return 200 with auth_token"
    negative: "POST /auth with valid credentials MUST NOT return 4xx or empty body"
  - id: A2
    positive: "POST /auth with invalid credentials MUST return 401 without auth_token"
    negative: "POST /auth with invalid credentials MUST NOT return 200 or auth_token"
constraints:
  - "MUST NOT store plain-text passwords"
  - "auth_token MUST be JWT with 24h TTL"
file_scope:
  - src/auth/handler.ts
  - src/auth/validator.ts
# --- END SPEC ---
```

Здесь две детали, которые стоит отметить особо.

**Positive + Negative assertions** — каждое условие содержит обе стороны: что *должно* произойти и что *не должно*. Это как двойная запись в бухгалтерии: логическая дыра, которую не замечаешь при формулировке одного условия, становится очевидной при написании противоположного.

**`file_scope`** — исчерпывающий список файлов, которые Implementer *имеет право изменить*. Шаг за пределы этого списка означает, что спецификация неверна и требует пересмотра. Это структурная изоляция реализации: агент не может тихо расширить scope задачи.

Формат assertions близок к acceptance criteria, знакомым по пользовательским историям — но здесь это не документация, а машиночитаемые условия, которые проверяются автоматически на каждом цикле.

### Каждый агент видит только своё

Один spec packet — разные представления для разных ролей. Вот где CE и SDD пересекаются особенно красиво: это Context Isolation, применённая к самой спецификации.

| Агент | Что получает из spec | Почему именно это |
|---|---|---|
| **Planner** | version + intent + title | *Пишет* assertions сам — если получит чужие, будет их копировать |
| **Implementer** | Полный spec packet | Нужно всё: что делать, в каких файлах, какие ограничения |
| **Reviewer** | intent + assertions + file_scope + evidence | Верифицирует выполнение, не перепроверяет намерение |
| **Tester** | assertions + file_scope + constraints | Проверяет соответствие, не интерпретирует замысел |

Planner, который не видит assertions, не будет их копировать — он их *формулирует сам*. Tester, который не видит intent, не будет его переосмыслять — он его *верифицирует*. Разные роли, разная ответственность, разный контекст.

### Auto-feedback loop: как SDD использует механизмы CE

CE ввела auto-feedback loop как механизм — цикл self-correction в рамках заданного числа итераций с circuit breaker на трёх. SDD наполняет этот механизм содержанием: теперь каждая итерация проверяет не просто «качество кода», а соответствие assertions из spec packet.

![Петля верификации в SDD: implementer реализует, reviewer проверяет по spec, при необходимости цикл повторяется — максимум три итерации, затем эскалация к человеку.](Illustrations/Ralph%20loop%20for%20subagents.png)

Цикл объединяет два уровня:
- Reviewer проверяет код по динамически загруженным **coding standards** — тем самым, которых не было в war story из Layer 0 про `any` везде
- Tester проверяет GIVEN/WHEN/THEN из spec packet — «готово» означает: все assertions прошли, не только «код компилируется»

При этом Tester в SDD-режиме работает в трёх режимах, и это важно. **Standard execution** — запуск тест-сьюта с созданием bug-задачи на каждый фейл. **Assertion execution** — независимая верификация claims Implementer'а: Tester переходит к конкретному `file:line` и проверяет, соответствует ли реальный код assertion'у. Если Implementer написал «A1: PASS», а код этого не делает — Tester поймает расхождение. Это закрывает уязвимость самооценки: агент не может объективно оценить собственную работу. **Integration verification** — проверка кросс-задачных взаимодействий после завершения всей фичи. Три режима — три уровня уверенности, каждый видит то, что предыдущий не может.

### Прогрессивное внедрение

Не нужно внедрять всё сразу. SDD прогрессивный по природе:

- **Slice 1** — базовые assertions + определение готовности. Минимальная ценность уже здесь.
- **Slice 2** — трекер фич (`feature-tracker.json`): реестр, который помнит состояние каждой фичи между сессиями. Возобновить работу без потери состояния — в любой момент.
- **Slice 3** — governance: `constitution.md` (неизменяемые принципы проекта), двухуровневая верификация, `failure-patterns.md` — чтобы система не повторяла одни и те же просчёты.
- **Slice 4** — масштабирование: spec-шаблоны (REST CRUD, auth flow), расширенные агенты, метрики. В roadmap.

**CE — фундамент**, применимый к любому проекту с AI-ассистентом, её принципы работают даже при одиночном агенте. **SDD — следующий слой**, специфичный для проектов, где «готово» должно быть строго верифицируемым. Не каждому проекту нужен Slice 4. Но каждому проекту нужна хотя бы одна строчка с определением готовности.

## Шаблон: все слои в одной коробке

`context-engineering-template` — готовая реализация всех описанных принципов для Claude Code. Не application code — только оркестрация: определения агентов, правила координации, файловые конвенции, quality gates. Клонируете в любой репозиторий — получаете рабочий workflow с шестью специализированными агентами.

Ключевые файлы, которые стоит изучить в первую очередь: `CLAUDE.md` (dispatch loop, ≤200 строк), `.claude/agents/` (шесть агентов), `.claude/skills/` (spec-protocol, coding-standards, review-checklist), `coding-standards-sources.yaml` (реестр стандартов по языкам).

**Dynamic Coding Standards.** При первом запуске шаблон сканирует репозиторий: находит `package.json`, `requirements.txt`, `go.mod` — по обнаруженному стеку автоматически загружает нужные стандарты и применяет их к Implementer-агенту через проверенную иерархию (override → verified → community). War story из Layer 0 про `any` везде — именно этот механизм его предотвращает. Новый язык в стеке — добавьте запись в конфиг.

**Измеримые параметры системы:**

| Метрика | Значение | Что это означает |
|---|---|---|
| Целевой токен-бюджет | < 128k | Весь проект — в одной сессии |
| Порог автокомпакции | ~80k | Предсказуемое управление памятью |
| Экономия через local-first RAG | 60–80% | Меньше внешних запросов |
| Максимум циклов auto-feedback loop | 3 | Эскалация вместо бесконечного цикла |
| Декомпозиция задач | 3–5 файлов | Управляемые единицы работы |
| Размер CLAUDE.md | ≤200 строк | Ядро само по себе компактно |

**Честно о сильных сторонах и ограничениях.** Stateless архитектура делает любой сбой восстановимым через git-историю. Self-discovering агенты означают: добавить специалиста — создать файл. Delegated Mechanism Selection позволяет добавлять агентов без изменения оркестратора. Прогрессивная SDD — можно начать со Slice 1 и дорасти до Slice 3.

При этом шаблон — не plug-and-play. После клонирования нужно заполнить три файла с плейсхолдерами (`review-checklist.md`, `testing-strategy.md`, `architecture-principles.md`): без этого агенты работают с пустыми правилами. Каждое решение требует записи в файл — это дисциплина, а не автоматика. Для одноразового скрипта или задачи на вечер шаблон — оверкилл. Slice 4 компоненты пока в roadmap.

Матрица применимости простая: проект длиннее одного рабочего дня, больше двух параллельных задач, нужен code review — шаблон оправдан.

## Два пути вперёд

Если вы дочитали до этого места, скорее всего уже примеряете описанное к своей работе. Обычно это выглядит одним из двух способов.

### Путь A: взять шаблон как есть

```bash
npx degit vospr/context-engineering-template my-project
cd my-project
git init
```

*(Актуальный путь к репозиторию — в README шаблона.)*

Обязательно заполните три файла после клонирования:

- `.claude/skills/review-checklist.md` — ваши критерии code review
- `.claude/skills/testing-strategy.md` — ваш подход к тестированию
- `.claude/skills/architecture-principles.md` — ваши архитектурные ограничения

`coding-standards.md` кастомизировать не нужно — это dynamic loader, работает автоматически.

### Путь B: взять идеи в свой проект

Если у вас уже есть рабочий процесс с Claude Code, Copilot или другим AI-ассистентом, вот что стоит перенести в первую очередь — примерно в таком порядке отдачи:

**Заведите `decisions.md`.** Файл, куда сразу пишутся архитектурные решения. Одно это изменение убирает половину проблем из Layer 0. Добавьте привычку делать git commit после каждой значимой задачи агента — страховочная сеть встроена в процесс.

**Настройте dispatch loop в `CLAUDE.md`.** Даже упрощённая версия из 50 строк — читать состояние → выбрать задачу → диспатчировать → записать результат — даёт предсказуемость, которой нет при свободном диалоге с агентом.

**Добавьте `coding-standards-sources.yaml`** для вашего стека. Агент перестанет угадывать, что считается хорошим кодом в вашем проекте.

**Введите явный review-цикл с circuit breaker**: implementer → reviewer → максимум 3 итерации → эскалация к человеку. Без этого ограничения агент может итерировать бесконечно.

**Попробуйте SDD в минимальном варианте.** Три GIVEN/WHEN/THEN к каждой задаче перед реализацией. Занимает пять минут — устраняет половину разногласий о том, готова ли фича.

**Разделите роли** в инструкциях агентам: не один универсальный системный промпт, а отдельные для researcher / implementer / reviewer. Каждый получает только свой контекст.

---

В 2026 году разница между командами, которые эффективно работают с AI, и теми, кто застрял в режиме умного автодополнения, определяется не выбором модели. Она определяется тем, насколько осознанно выстроена работа с контекстом.

Context Engineering — это дисциплина. Spec-Driven Development — договорённость о результате. Вместе они превращают одиночного AI-ассистента в систему, которой можно доверять длинные, многосессионные, параллельные задачи.

Начните с малого: заведите `decisions.md`, добавьте три assertion к следующей задаче, разделите роли. Это не требует ни шаблона, ни специального инструмента — только решения делать это системно.
